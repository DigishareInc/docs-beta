Étendez vos contacts (Thirds) et autres entités avec des structures de données personnalisées adaptées à votre logique métier. Digishare prend en charge l'indexation dynamique des métadonnées, vous permettant de stocker, de vérifier le type et de filtrer les propriétés personnalisées à la volée.

Nous proposons deux approches en fonction de vos besoins spécifiques : une approche rapide sans schéma et une approche avancée via l'API avec un schéma strictement typé.

---

## Approches Architecturales

::card-group
::card{title="Champs Rapides (Sans schéma)" icon="i-mdi-flash"}
Stockez des paires clé-valeur directement dans l'objet JSON générique `data.custom_fields` de l'entité. Les champs sont automatiquement déduits, synchronisés et implicitement indexés par le système de Champs Personnalisés lors de la création ou de la mise à jour de l'entité. Idéal pour les métadonnées non structurées ou changeant rapidement.
::

::card{title="API Avancée (Schéma Strict)" icon="i-mdi-database"}
Définissez explicitement des schémas de données à l'aide de l'API dédiée aux Définitions de Champs Personnalisés. Cela vous permet d'imposer un typage strict (`string`, `float`, `date`, `bool`, etc.), de définir des options d'énumération (`options`) et de segmenter les champs de manière sécurisée dans des groupes (`groups`). Idéal pour l'analytique, les intégrations systématiques et le filtrage complexe.
::
::

## L'Approche Rapide : `data.custom_fields`

Stockez des données personnalisées de manière implicite lors des opérations standard sur les entités. Les processus d'arrière-plan de Digishare détecteront automatiquement les nouvelles clés, créeront des définitions de schéma implicites limitées au contexte de votre entreprise et indexeront les valeurs pour des requêtes haute performance.

### 1. Création lors de la création de l'entité

Fournissez l'objet `data.custom_fields` lors de la création d'un Contact (Third).

::api-playground
---
method: POST
url: "https://api.digishare.ma/v1/thirds"
headers:
  Authorization: "Bearer VOTRE_ACCESS_TOKEN"
  Content-Type: "application/json"
body:
  first_name: "Jean"
  last_name: "Dupont"
  data:
    custom_fields:
      solde: 150.75
      niveau_fidelite: "gold"
      est_vip: true
responseSample:
  id: "third_123"
  status: "created"
---
::

### 2. Mise à jour des champs d'une entité existante

Modifiez le payload de données de l'entité pour fusionner de nouveaux champs ou écraser les champs personnalisés existants.

::api-playground
---
method: PATCH
url: "https://api.digishare.ma/v1/thirds/{id}"
variables:
  id: "ID_DU_THIRD"
headers:
  Authorization: "Bearer VOTRE_ACCESS_TOKEN"
  Content-Type: "application/json"
body:
  data:
    custom_fields:
      solde: 200.0
      dernier_achat: "2026-01-18"
responseSample:
  status: "updated"
---
::

## L'Approche Avancée : Schémas Explicites

Pour les implémentations de niveau entreprise, vous pouvez définir explicitement le schéma et les propriétés de vos champs personnalisés avant d'y injecter des données. Cette approche garantit la sécurité du typage et évite la prolifération de données disparates.

### 1. Définir ou gérer les schémas de champs

Utilisez l'API des définitions pour dicter comment les champs personnalisés doivent être traités. Les définitions de champs sont automatiquement limitées et sécurisées au contexte de votre entreprise authentifiée.

**Propriétés :**

- `key` (string) : L'identifiant unique du champ (ex: `solde_compte`).
- `label` (string) : Nom lisible par l'homme (ex: `Solde du Compte`).
- `type` (string) : Le type de données (voir les types supportés ci-dessous).
- `group` (string, optionnel) : Groupement logique (par défaut : `default`).
- `model_type` (string, optionnel) : L'entité à laquelle ce champ appartient (par défaut : `third`).
- `options` (array, optionnel) : Valeurs scalaires autorisées pour les types énumérés.

::api-playground
---
method: POST
url: "https://api.digishare.ma/v1/custom-field-definitions"
headers:
  Authorization: "Bearer VOTRE_ACCESS_TOKEN"
  Content-Type: "application/json"
body:
  key: "programme_fidelite"
  label: "Programme de Fidélité"
  type: "string"
  group: "marketing"
  model_type: "third"
  options: ["Aucun", "Silver", "Gold", "Platinum"]
responseSample:
  id: "def_456"
  status: "created"
---
::

::api-playground
---
method: GET
url: "https://api.digishare.ma/v1/custom-field-definitions"
headers:
  Authorization: "Bearer VOTRE_ACCESS_TOKEN"
responseSample:
  data:
    - key: "programme_fidelite"
      label: "Programme de Fidélité"
      type: "string"
---
::

**Mettre à jour une définition :**

::api-playground
---
method: PATCH
url: "https://api.digishare.ma/v1/custom-field-definitions/{id}"
variables:
  id: "ID_DEFINITION"
headers:
  Authorization: "Bearer VOTRE_ACCESS_TOKEN"
  Content-Type: "application/json"
body:
  label: "Programme de Fidélité Mis à Jour"
  options: ["Aucun", "Silver", "Gold", "Platinum", "Diamond"]
responseSample:
  status: "updated"
---
::

**Supprimer une définition :**

::api-playground
---
method: DELETE
url: "https://api.digishare.ma/v1/custom-field-definitions/{id}"
variables:
  id: "ID_DEFINITION"
headers:
  Authorization: "Bearer VOTRE_ACCESS_TOKEN"
responseSample:
  status: "deleted"
---
::

> [!WARNING]
> La suppression d'une définition retire l'application du schéma et les règles de typage. Toutes les données existantes dans vos contacts resteront mais seront traitées comme des "Champs Rapides" non structurés.

#### Types de Données Supportés

| Type     | Stockage Interne   | Comportement d'Auto-Casting             |
| -------- | ------------------ | --------------------------------------- |
| `string` | Texte Indexé       | Stocké tel quel                         |
| `int`    | Entier Natif       | Formaté en entier                       |
| `float`  | Entier (x1000)     | Formaté en nombre à virgule             |
| `bool`   | Booléen / Bit      | `1/0`, `true/false`, `yes/no`           |
| `date`   | Entier (`Ymd`)     | Analyse ISO `YYYY-MM-DD` ou `DD/MM/YYYY`|

### 2. Modifier les champs personnalisés d'une entité directement

Au lieu de modifier l'intégralité du Third, ciblez les champs personnalisés directement via l'endpoint dédié.

::api-playground
---
method: POST
url: "https://api.digishare.ma/v1/thirds/{id}/custom-fields"
variables:
  id: "ID_DU_THIRD"
headers:
  Authorization: "Bearer VOTRE_ACCESS_TOKEN"
  Content-Type: "application/json"
body:
  fields:
    solde: 150.75
    nombre_commandes: 12
  group: "facturation"
responseSample:
  status: "success"
---
::

### 3. Récupérer les champs personnalisés d'une entité

Récupérez tous les champs personnalisés attachés, indépendamment de la représentation de l'entité parente.

::api-playground
---
method: GET
url: "https://api.digishare.ma/v1/thirds/{id}/custom-fields"
variables:
  id: "ID_DU_THIRD"
headers:
  Authorization: "Bearer VOTRE_ACCESS_TOKEN"
responseSample:
  data:
    solde: 150.75
    nombre_commandes: 12
    programme_fidelite: "Gold"
---
::

### 4. Supprimer des champs spécifiques

Effacez des clés personnalisées spécifiques du stockage de données de l'entité.

::api-playground
---
method: DELETE
url: "https://api.digishare.ma/v1/thirds/{id}/custom-fields"
variables:
  id: "ID_DU_THIRD"
headers:
  Authorization: "Bearer VOTRE_ACCESS_TOKEN"
  Content-Type: "application/json"
body:
  keys: ["solde", "nombre_commandes"]
responseSample:
  status: "removed"
---
::

## Indexation et Filtrage Avancés

Digishare indexe les champs personnalisés nativement, offrant des requêtes dynamiques puissantes capables d'effectuer des recherches par plage et relationnelles sur votre ensemble de données.

### Filtrer les requêtes par champs personnalisés

Trouvez des entités correspondant à des contraintes de métadonnées. Les opérateurs standard fonctionnent efficacement sur les types `int`, `float` et `date`.

```bash
# Obtenir les contacts dépassant un solde spécifique
GET /v1/thirds?custom_filters={"solde":{"operator":">","value":100}}

# Obtenir les contacts intégrés après une date spécifique
GET /v1/thirds?custom_filters={"dernier_achat":{"operator":">=","value":"2026-01-01"}}

# Obtenir les contacts VIP où le booléen est exactement vrai
GET /v1/thirds?custom_filters={"est_vip":{"operator":"=","value":true}}
```

#### Référence des Opérateurs

| Opérateur | Comparaison |
| -------- | ---------- |
| `=`      | Égal (Défaut implicite) |
| `>`      | Strictement Supérieur à |
| `<`      | Strictement Inférieur à |
| `>=`     | Supérieur ou Égal |
| `<=`     | Inférieur ou Égal |
| `!=`     | Différent de |

## Imports en Masse et Ingestion

Lors de l'importation de données en masse via des **Tableurs (CSV/Excel)**, l'**API Bulk standard** ou l'**API d'Ingestion (Egressor)**, Digishare utilise une structure plate avec un protocole de préfixe pour distinguer les champs standard des métadonnées personnalisées.

### Le Protocole de Préfixe

Contrairement à l'API pour une entité unique où vous imbriquez les champs sous `data.custom_fields`, les opérations en masse nécessitent d'ajouter le préfixe `cf_` ou `custom_` aux clés/colonnes. Le moteur retirera le préfixe et liera les valeurs en tant que champs personnalisés.

**Méthodes en masse supportées :**
1. **Tableurs** : En-têtes de colonnes (ex: `cf_solde`).
2. **API Bulk JSON** : Clés d'objets dans le tableau `data` (ex: `"cf_solde": 150.75`).
3. **API d'Ingestion** : Clés d'objets dans le payload de l'événement (ex: `"cf_solde": 150.75`).

| Nom de Clé / Colonne | Slug Final | Logique |
| ----------------- | ---------- | ----- |
| `cf_solde`        | `solde`    | Préfixé (Auto-indexé) |
| `custom_vip`      | `vip`      | Préfixé (Auto-indexé) |
| `email`           | `email`    | Champ Standard |
